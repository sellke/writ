# Technical Spec: Git-Notes Audit Channel

> Parent: [`../spec.md`](../spec.md)

## 1. Notes ref & git mechanics

- **Ref:** `refs/notes/writ` (dedicated). All operations pass `--ref=writ` explicitly.
- **Attach:** `git notes --ref=writ add -F <tmpfile> <sha>` (use `-f` to overwrite if re-shipping the same SHA).
- **Read:** `git log --notes=writ` or `git notes --ref=writ show <sha>`.
- **Sync refspecs** (added to `.git/config`, idempotent):
  - Fetch: `+refs/notes/writ:refs/notes/writ`
  - Push: `refs/notes/writ`
- **Rewrite copying (optional hardening):** `git config notes.rewriteRef refs/notes/writ` so `git rebase`/`git commit --amend` copy Writ notes forward on *local* rewrites. Does **not** rescue squash-merge (new commit, not a rewrite of one) — which is why the primary attach point is post-land.

## 2. Attach-point resolution

### `/ship`

`/ship` already performs the merge/rebase onto the base branch. After the land step:

1. Capture the **landed SHA**:
   - Squash-merge → the single squash commit (`git rev-parse <base>` after merge).
   - Merge commit → the merge commit SHA.
   - Rebase-and-merge → the **tip** of the replayed commits on base.
2. Resolve the **spec folder** being shipped (from the branch/PR context `/ship` already tracks).
3. Compose the digest (§3) and attach to the landed SHA.
4. If a note already exists on that SHA under `refs/notes/writ`, overwrite (`add -f`) — a re-ship supersedes.

### `/release`

After tagging the release commit:

1. Determine specs shipped since the previous release tag (`/release` already assembles the changelog from completed specs).
2. Compose the version rollup (§4).
3. Attach to the tag's target commit under `refs/notes/writ`.

## 3. Spec-level digest schema (attached at `/ship`)

Composed from each completed story's `## What Was Built` record in the spec folder.

```
Writ Audit Digest (spec)
========================
Spec:     {spec-folder-id} — {spec title}
Landed:   {base-branch} @ {landed-sha}
Source:   {source-range, e.g. abc123..def456} ({N} commits)
Shipped:  {YYYY-MM-DD} by {git user.name}

Stories
-------
| Story | Review | Drift | Coverage |
|-------|--------|-------|----------|
| story-1-... | PASS | None | 86% |
| story-2-... | PASS | Small (DEV-003) | 91% |

Aggregate
---------
Verdict:   {PASS | PASS w/ drift | DEGRADED}
Drift:     highest = {None|Small|Medium|Large}; DEV-IDs: {DEV-003, ...}
Coverage:  {aggregate/na}
Files:     {C} created, {M} modified
Review:    {total review iterations across stories}

Refs
----
spec:      .writ/specs/{spec-folder-id}/spec.md
drift-log: .writ/specs/{spec-folder-id}/drift-log.md (if present)

Generated by Writ {VERSION}
```

**Composition rules:**
- Parse each `story-*.md` for its `## What Was Built` → `### Review Outcome` (Result, Drift, iteration count), `### Test Results` (Coverage), `### Files Created/Modified`, `### Deviations from Spec` (DEV-IDs).
- Aggregate verdict = worst story result; drift = highest severity present; DEV-IDs = union.
- **Never** copy narrative "Implementation Decisions" prose verbatim if it risks leaking transcript-like content — include only the audit fields above.

### Fallback / nil input

If no WWB records are found (e.g., `--quick` spec, or manual work):

```
Writ Audit Digest (spec, minimal)
=================================
Spec:     {spec-folder-id or "(none)"}
Landed:   {base} @ {landed-sha}
Files:    {git diff --stat summary}
Note:     No "What Was Built" records found — minimal digest.
Generated by Writ {VERSION}
```

## 4. Version rollup schema (attached at `/release`)

```
Writ Release Audit — v{VERSION}
===============================
Released: {YYYY-MM-DD}
Tag:      {tag} @ {tag-target-sha}
Previous: v{prev-version}

Specs shipped this release
--------------------------
- {spec-folder-id} — {title} — {aggregate verdict, highest drift}
- ...

Changelog: CHANGELOG.md#{version-anchor}
Generated by Writ {VERSION}
```

## 5. Opt-out mechanism

- **Flag location:** a single source of truth readable by `ship.md`, `release.md`, and `install.sh`. Recommended: an env/config marker documented in `git-notes-audit-format.md` (e.g., a `writ.auditNotes` git config key: `git config writ.auditNotes false`).
  - Rationale for a git-config key: it is per-repo, travels in `.git/config`, and is trivially checkable by commands (`git config --bool writ.auditNotes`, default true).
- **When disabled:**
  - `/ship` and `/release` skip note composition/attachment entirely (silent).
  - `install.sh` skips adding the refspecs, and (if run with the disable intent) removes any Writ-added refspecs it previously created.
- **Opt-out must leave no residue:** no `refs/notes/writ` refspec lines, no dangling config beyond the single `writ.auditNotes=false` marker.

## 6. `install.sh` configuration (guarded)

Pseudologic:

```
if audit notes enabled (writ.auditNotes != false):
    remote = default push remote (origin fallback)
    if fetch refspec "+refs/notes/writ:refs/notes/writ" not already present:
        git config --add remote.$remote.fetch "+refs/notes/writ:refs/notes/writ"
    if push refspec "refs/notes/writ" not already present:
        git config --add remote.$remote.push "refs/notes/writ"
else:
    remove any Writ-added refs/notes/writ refspecs (idempotent)
```

- Idempotent: never add duplicate refspecs (grep existing config first).
- No-op gracefully when not in a git repo or no remote configured (warn, skip).

## 7. `/status` read surface

Add one line to the status output when at least one `refs/notes/writ` note exists:

```
📝 Last audit note: {short-sha} — {spec title} ({date})
```

Resolve via `git log --notes=writ -1 --format=...` filtered to commits carrying a Writ note. If none, omit the line.

## 8. Eval check

Add a static check (`scripts/eval.sh` + a small `eval-*` helper) asserting:
- `commands/ship.md` and `commands/release.md` reference `refs/notes/writ`.
- Both reference the non-blocking rule (audit failure never blocks ship/release).
- `install.sh` guards the refspec config behind the opt-out marker.

## 9. Non-goals / explicit exclusions

- No per-story branch-local notes.
- No bespoke reader command — `git log --notes=writ` is canonical.
- No historical backfill.
- No PR-comment mirroring.
