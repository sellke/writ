# Technical Spec: Product Reconciliation

> Source: .writ/specs/2026-07-11-product-reconciliation/spec.md

## Nature of the change

All three deliverables are **markdown instruction edits to existing command
files** — no scripts, no eval changes. `/verify-spec`, `/plan-product`, and
`/retro` are agent-executed command docs; adding a mode means adding instructions
the agent follows, consistent with how these commands already work.

## Story 1 — `commands/verify-spec.md`: `--product` mode

**Modes table:** add a row:

| `/verify-spec --product` | Product docs | Run the Product Consistency check set over `.writ/product/` + `.writ/context.md`; hybrid auto-fix |

**New section — "Product Consistency Checks (`--product`)":** a self-contained
check set, explicitly *not* checks 1–7. Documents inputs, the four checks, and
the auto-fix vs report-only split.

Inputs read:
- `.writ/product/mission.md` (authoritative vision + Key Features phases)
- `.writ/product/roadmap.md` (authoritative phase statuses)
- `.writ/product/mission-lite.md` (derivative)
- `.writ/context.md` (derivative, regenerated by `/status` Step 8)
- `.writ/decision-records/adr-*.md` (for reference resolution)

Check dispositions:

| # | Check | Disposition |
|---|---|---|
| 1 | Phase-status parity (mission ↔ roadmap) | report-only |
| 2 | ADR reference resolution | report-only |
| 3 | Derivative freshness (mission-lite, context) | auto-fix: regenerate |
| 4 | Shipped-claim sanity (heuristic) | report-only |

**Auto-fix mechanics:** reuse verify-spec's existing spec-lite regeneration
pattern (Phase 4.4) — regenerate `mission-lite.md` from `mission.md` and
`.writ/context.md` from the `/status` Step 8 schema. Never rewrite authoritative
`mission.md`/`roadmap.md` prose; authoritative divergence is reported for a human.

**Boundary note:** add a sentence pairing `--product` (consistency, before) with
`/plan-product --reconcile` (revision, after), cross-referencing it.

**Report:** reuse the existing verification report shape; the check table for
`--product` shows the 4 product checks instead of the 7 spec checks. Write to
`.writ/product/verification-YYYY-MM-DD.md` (product-scoped, not per-spec).

## Story 2 — `commands/plan-product.md`: `--reconcile` posture

Add an invocation/mode section near the top:

**`--reconcile` flow** (replaces greenfield Phase 1 discovery):
1. **Scan existing** `.writ/product/` (mission, roadmap, mission-lite) + ADRs.
2. **Diff vs reality:** enumerate shipped specs under `.writ/specs/` (status
   Complete), compare roadmap phase statuses to what actually shipped, scan git
   for recent direction; identify stale phase labels, unshipped "next" items that
   shipped, and roadmap items with no evidence.
3. **Propose targeted revisions** in Plan Mode — edits to mission/roadmap and
   *new* ADRs only for genuine direction changes. Do **not** regenerate the whole
   product package from scratch.
4. On lock, apply the revisions (Agent Mode, AskQuestion gate as in greenfield).

**Boundary note:** state that `--reconcile` revises (after deciding to change),
while `/verify-spec --product` only checks consistency (before). Suggest running
`--product` first to see *what* drifted, then `--reconcile` to decide *what to do*.

## Story 3 — `commands/retro.md`: product-drift nudge

Add a step modeled **exactly** on Step 5.5 (Knowledge Consolidation Nudge):
read-only, mutates nothing, skips gracefully when `.writ/product/` is absent.

**Signal (cheap, observable):**
- A `roadmap.md` phase marked complete/shipped with a date **after**
  `mission.md`'s `Last Updated`, **or**
- A mission Key-Features phase label ("next"/planned) that contradicts the
  roadmap's status for that phase.

When a signal is present, print a one-line advisory, e.g.:

```
Product docs may be drifting — roadmap shows Phase N complete but mission still
frames it as upcoming. Consider `/verify-spec --product` to see the drift, or
`/plan-product --reconcile` to revise.
```

When no signal (or `.writ/product/` absent), print nothing. Never blocks the retro.
