# Story 3: "What Was Built" Completion Record

> **Status:** Completed ✅
> **Priority:** High
> **Dependencies:** None

## User Story

**As a** developer reading a completed story
**I want** a record of what was actually built — not just what was planned
**So that** future agents and developers have ground-truth context about implementation decisions, not just the original plan

## Acceptance Criteria

- [x] Given all gates pass for a story, when Step 4 (Story Completion) runs, then a `## What Was Built` section is appended to the story file before the commit
- [x] Given the coding agent reported deviations and the drift log has entries, when the "What Was Built" section is generated, then it references specific DEV-XXX deviation IDs
- [x] Given the coding agent made implementation decisions not in the original plan (e.g., chose a specific hook, added debounce), when the section is generated, then key decisions are captured from the coding agent's output
- [x] Given the story file already has all its standard sections, when `## What Was Built` is appended, then it appears after `## Definition of Done` and before any trailing content

## Notes

- The section is generated by the implement-story orchestrator, not by a subagent. It synthesizes information already available: coding agent output, drift log, and git diff stats.
- This implements the "system spec" concept from Behavior-Driven Prompting research — a record of implementation reality that survives alongside the original plan.
- Format example:
  ```
  ## What Was Built

  **Implemented:** 2026-03-15
  **Files:** 5 created, 3 modified

  **Key decisions made during implementation:**
  - Used `useLocalStorage` hook instead of context (simpler, story-scoped state)
  - Added debounce to search input (not in spec, but UX required it)

  **Deviations from plan:** See drift-log.md DEV-003, DEV-004
  ```
- Only touches `commands/implement-story.md`.
