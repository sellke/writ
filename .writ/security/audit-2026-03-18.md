## Medium Findings (Action Recommended)

### [SEC-001] Review Agent: 3 Security Check Items Dropped in Condensation

- **Severity:** Medium
- **Category:** Guardrail Regression
- **Location:** `agents/review-agent.md` — Section 3 (Security)
- **Description:** The refactoring condensed the 10-item security checklist into a single principle line. While 7 of 10 items are explicitly preserved in the condensed form, **3 items were dropped entirely:**

  1. **CORS/CSP headers** — `CORS/CSP headers appropriate (if applicable)` → not mentioned
  2. **File upload validation** — `File uploads validated (type, size) if applicable` → not mentioned
  3. **Sensitive data in logs/errors** — `Sensitive data not exposed in logs, error messages, or API responses` → not mentioned

- **Impact:** AI agents following the condensed instructions may not check for these three attack vectors during code review. CORS misconfiguration is a common vulnerability in web apps. File upload validation is critical for apps handling user files. Data leakage through logs/errors is a frequent finding in real audits.
- **Mitigating factor:** The addition of **"Security is never Minor — any security issue is at least Major severity"** is a quality bar that didn't exist before, which partially compensates by raising the review agent's overall security posture.
- **Remediation:** Add the 3 dropped items back to the condensed line. This adds ~20 words and zero structural complexity:

```markdown
### 3. Security
Input validation/sanitization, parameterized queries, XSS/injection prevention, auth checks,
no hardcoded secrets, dependency vulnerability check, CORS/CSP headers, file upload validation
(type and size), sensitive data not exposed in logs or error responses.
**Security is never Minor** — any security issue is at least Major severity.
```

### [SEC-004] .gitignore: Minimal Coverage

- **Severity:** Low
- **Category:** Infrastructure
- **Location:** `.gitignore`
- **Description:** The `.gitignore` only covers `.DS_Store` and `.writ/state/`. This is appropriate for a markdown-only project with no secrets, but if contributors ever add test fixtures, environment files, or key material, there's no safety net.
- **Impact:** Minimal for current project shape. Risk increases if Writ gains executable components.
- **Remediation:** Consider adding defensive patterns:

```gitignore
.env*
*.pem
*.key
credentials.json
```

### Shell Scripts
- **`set -euo pipefail`** used in all 3 scripts (install, migrate, update) — prevents silent failures
- **Temp directory cleanup** with `trap cleanup EXIT` — prevents orphaned clones
- **Self-dogfooding guard** — all scripts refuse to run on the Writ source repo itself
- **`rm -rf` usage** is scoped to temp directories and known paths only — no wildcard deletion
- **No arbitrary code execution** from user input — install.sh only copies/symlinks files from a known repository
- **Manifest integrity** — update.sh uses SHA-256 three-way diffing to prevent overwriting user customizations
